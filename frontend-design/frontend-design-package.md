## 2. 需求对齐校验报告

### 2.1 业务目标对齐

项目的前端目标不是做一个静态百科页，而是建立一个兼具科普性、探索性、对比性和可运营性的航空知识中枢。前端需要同时支撑：

1. 非专业用户快速理解航空器、事件、人物三类内容。
2. 用户以低学习成本完成搜索、筛选、浏览、对比和继续探索。
3. 内容运营人员通过后台完成内容录入、校验、发布与维护。
4. 通过智能问答、相关推荐、知识联动提升平台教育价值与展示亮点。

### 2.2 用户角色定义

| 角色 | 使用目标 | 核心任务 | 前端重点支持 |
| --- | --- | --- | --- |
| 学生/入门用户 | 快速看懂航空知识 | 搜索、查看详情、阅读推荐 | 低门槛表达、导览式 UI、术语解释 |
| 航空爱好者 | 深入比较不同机型与事件 | 精准筛选、多机型对比、关系跳转 | 高密度信息、表格式对比、快捷跳转 |
| 科普观众/评委 | 在短时间内理解作品价值 | 浏览首页、查看亮点、体验交互 | 强叙事首页、清晰演示路径、视觉记忆点 |
| 内容运营人员 | 高效录入和修订内容 | 新建条目、图片上传、字段校验、发布 | 后台表单、即时校验、版本状态、缺失提示 |
| 管理员 | 管控内容质量与风险 | 审核、下线、查看日志、监控质量 | 审计面板、异常提示、状态汇总 |

## 3. 前端技术架构方案

### 3.1 架构原则

1. 公共门户与后台管理共享设计系统与基础设施，减少样式和接口重复建设。
2. 以服务端渲染优化公开内容页面首屏、SEO 和分享能力。
3. 以后台高交互表单和前台丰富内容展示并存为前提，区分服务端数据与客户端交互状态。
4. 所有页面和模块支持后续扩展智能问答、时间轴、知识图谱和多语言能力。

### 3.2 技术选型

| 维度 | 方案 | 选型原因 |
| --- | --- | --- |
| 前端框架 | Next.js 15 + React 19 + TypeScript | 同时满足 SSR、SEO、路由组织、组件化和工程化要求 |
| 构建工具 | Next Build / Turbopack | 与 Next.js 原生整合，开发体验和构建效率更高 |
| 样式方案 | Tailwind CSS + CSS Variables + 局部 CSS Modules | 便于设计系统落地、主题控制与细粒度页面定制 |
| 组件体系 | Radix UI + shadcn/ui 二次设计封装 | 提供可访问基础能力和高可定制性 |
| 状态管理 | TanStack Query + Zustand | 服务端状态与本地交互状态分治，易维护 |
| 表单方案 | React Hook Form + Zod | 后台复杂表单性能稳定，校验规则可复用 |
| 请求层 | Axios 或 Fetch Wrapper | 支持拦截器、统一错误处理、重试与缓存策略 |
| 图表可视化 | ECharts | 适合时间轴、统计和信息图展示 |
| 测试体系 | Vitest + Testing Library + Playwright + axe-core | 覆盖单元、组件、E2E 与无障碍测试 |

### 3.3 路由策略

采用单仓单应用模式，在同一 Next.js 项目中通过 Route Groups 区分前台与后台：

```text
app/
  (public)/
    page.tsx
    search/page.tsx
    aircraft/page.tsx
    aircraft/[slug]/page.tsx
    compare/page.tsx
    events/page.tsx
    events/[slug]/page.tsx
    persons/page.tsx
    persons/[slug]/page.tsx
  (admin)/
    admin/login/page.tsx
    admin/dashboard/page.tsx
    admin/aircraft/page.tsx
    admin/aircraft/new/page.tsx
    admin/aircraft/[id]/edit/page.tsx
    admin/events/page.tsx
    admin/persons/page.tsx
    admin/audit-logs/page.tsx
```

路由策略说明：

1. 公开内容页优先使用服务端数据获取与静态增量缓存。
2. 后台页统一走客户端鉴权与细粒度权限守卫。
3. 对比页、搜索页、问答抽屉通过 URL Query 保持可分享和可回放。

### 3.4 状态管理分层

| 状态类型 | 管理方式 | 适用场景 |
| --- | --- | --- |
| 服务端数据 | TanStack Query | 列表、详情、对比数据、推荐数据 |
| 会话状态 | HttpOnly Cookie + 中间件 | 后台登录态、鉴权跳转 |
| UI 状态 | Zustand | 侧边栏、问答抽屉、对比栏、全局提示 |
| 表单状态 | React Hook Form | 后台内容录入、筛选表单 |
| URL 状态 | Search Params | 搜索条件、排序方式、对比项 |

### 3.5 模块化拆分方式

```text
src/
  components/
    base/
    layout/
    aircraft/
    events/
    persons/
    compare/
    ai-assistant/
    admin/
  features/
    search/
    recommendation/
    content-validation/
  lib/
    api/
    auth/
    cache/
    analytics/
    accessibility/
  store/
  hooks/
  types/
  styles/
```

模块职责说明：

1. `components` 负责可复用视图层。
2. `features` 负责与业务强绑定的组合模块。
3. `lib/api` 负责接口契约、请求封装、错误模型和缓存策略。
4. `types` 与后端 DTO 保持一一对应，避免字段漂移。

### 3.6 跨端适配方案

采用同一套响应式 Web 方案覆盖三档终端：

- Mobile：360px - 767px
- Tablet：768px - 1199px
- Desktop：1200px 以上

适配策略：

1. 信息导航在移动端折叠为底部主导航 + 顶部搜索入口。
2. 对比表在移动端切换为“分段卡片对比”而非横向大表。
3. 后台复杂表单在移动端不作为主使用场景，只保证可访问与最小编辑能力。
4. 图片、图表和时间线组件支持断点裁切和懒加载。

| `react-hook-form` | 表单性能优化 | 后台表单统一接入 |

## 4. UI 视觉设计方案

### 4.1 品牌方向

视觉基调定义为“夜航档案馆”：

1. 不是普通科技蓝网页，而是兼具航空仪表精度感和博物馆档案质感。
2. 首页强调探索感与知识纵深，后台强调稳定、清晰和高效率。
3. 公共前台呈现“飞行文明”和“知识图谱”的气质，避免纯工具站观感。

### 4.2 设计系统

#### 色彩规范

| 色板 | 用途 | 色值 |
| --- | --- | --- |
| Flight Navy | 主背景/导航 | `#08111F` |
| Atlas Blue | 主品牌强调色 | `#1B4D8C` |
| Radar Cyan | 交互高亮/链接 | `#58D7FF` |
| Archive Sand | 卡片底/档案底色 | `#E7DCC4` |
| Signal Amber | 重点标注/告警 | `#FFB24D` |
| Lift Green | 成功状态 | `#63D297` |
| Warning Red | 错误反馈 | `#F25A5A` |
| Cloud White | 主要正文 | `#F6F7FB` |
| Slate Gray | 次级正文 | `#8D96A8` |

#### 字体层级

| 层级 | 字体建议 | 字重 | 说明 |
| --- | --- | --- | --- |
| Display | Noto Serif SC | 700 | 首页大标题、展陈区标题 |
| Heading | Noto Sans SC | 600 | 页面主标题、小节标题 |
| Body | Noto Sans SC | 400/500 | 正文、说明、参数 |
| Data | IBM Plex Mono | 500 | 参数、编号、状态标签、时间轴数据 |

#### 间距与圆角

1. 使用 4px 基准网格。
2. 页面大区块间距：64/80/96。
3. 卡片内边距：16/20/24。
4. 圆角体系：8、16、24，不使用过度圆润。

#### 阴影与质感

1. 使用低透明度蓝黑投影营造舱室深度。
2. 结合细网格、噪点和轻微金属边线，增强“档案馆 + 仪表盘”质感。
3. 强调信息分层而非炫技发光。

### 4.3 组件库定义

必备组件：

1. 全局导航栏
2. 搜索框与自然语言搜索提示
3. 分类筛选器
4. 航空器卡片
5. 参数数据卡
6. 对比表和差异高亮单元格
7. 事件时间轴卡片
8. 人物关系卡
9. 问答抽屉/对话面板
10. 后台数据表格
11. 后台编辑表单
12. 状态标签、风险提示、空状态和错误反馈

### 4.4 交互范式

1. 关键操作路径遵循“先结论、再展开”的阅读逻辑。
2. 高密度信息区域通过分组、吸顶导航和对比高亮降低阅读负担。
3. 智能助手默认作为“辅助入口”，不抢占主内容层级。
4. 后台表单强调即时校验和边填边预览。

### 4.5 核心页面设计范围

| 页面 | 设计重点 | 关键组件 |
| --- | --- | --- |
| 首页 | 叙事式入口、热门内容、智能助手入口 | Hero、搜索、分类卡、时间线、推荐区 |

| 对比页 | 参数对比和结论摘要 | 机型槽位、对比表、差异标记、总结卡 |
| 事件详情页 | 历史背景、影响和关联知识 | 时间线、关联卡、影响摘要 |
| 人物详情页 | 生平、贡献、关联作品 | 信息卡、关系图条目、相关推荐 |

### 4.6 无障碍设计标准

1. 全局对比度达到 WCAG 2.2 AA，正文文本不低于 4.5:1。
2. 所有可点击组件提供 `focus-visible` 样式和键盘导航路径。
3. 图像全部提供明确 `alt`，信息图提供文字补充说明。
4. 表单错误不仅用颜色提示，还提供图标和文本说明。
5. 所有弹层、抽屉、对话框具备语义角色与焦点管理。

### 4.7 响应式适配设计

| 断点 | 布局策略 |
| --- | --- |
| Mobile | 单列流式布局，保留搜索、内容卡、对比卡片模式 |
| Tablet | 双栏内容区，保持详情页和列表页效率 |
| Desktop | 三栏或 12 栏网格，强化探索型阅读与多面板展示 |

## 5. 交互流程设计

### 5.1 核心用户路径

#### 路径 A：入门用户探索

1. 进入首页
2. 使用主搜索框输入“喷气时代最有代表性的客机”
3. 进入搜索结果页
4. 点击某机型进入详情页
5. 查看参数和相关推荐
6. 打开“问问智能助手”继续追问

#### 路径 B：爱好者对比机型

1. 在列表页勾选 2-4 架机型
2. 进入对比页
3. 通过固定维度查看差异
4. 切换“仅看差异”
5. 阅读智能总结
6. 跳转到某一机型详情深挖

#### 路径 C：运营录入内容

1. 管理员登录后台
2. 进入航空器管理
3. 新建机型条目并上传图片
4. 输入参数、来源和摘要
5. 实时查看字段缺失与单位冲突提示
6. 预览发布并同步前台

### 5.2 页面跳转逻辑

| 来源页面 | 用户操作 | 目标页面 |
| --- | --- | --- |
| 首页 | 搜索关键词 | 搜索结果页 |
| 首页 | 点击热门机型 | 飞行器详情页 |
| 搜索页 | 勾选多机型并点击对比 | 对比页 |
| 飞行器详情页 | 点击关联事件/人物 | 事件详情页/人物详情页 |
| 任意公开页 | 点击问答入口 | 当前页问答抽屉 |
| 后台列表页 | 点击新建 | 后台编辑页 |
| 后台编辑页 | 点击预览 | 前台预览态 |

### 5.3 状态反馈规则

| 状态 | 反馈形式 | 规则 |
| --- | --- | --- |
| 加载中 | Skeleton + 轻动效 | 超过 400ms 才显示骨架，避免闪烁 |
| 空结果 | 说明文案 + 推荐入口 | 给出筛选建议和继续探索入口 |
| 成功保存 | Toast + 状态标签变更 | 保持用户上下文，不强制跳页 |
| 失败保存 | 顶部告警 + 字段级提示 | 先摘要错误，再精确定位字段 |
| 网络波动 | 横幅提示 + 重试按钮 | 对公开内容优先显示缓存内容 |
| 权限不足 | 403 页面 + 返回路径 | 告知原因与可申请权限方式 |

### 5.4 异常场景处理

| 异常场景 | 前端处理方案 |
| --- | --- |
| 网络错误 | 展示弱打扰横幅、支持重试、保留当前输入内容 |
| 接口超时 | Query 超时提示 + 局部重载，不清空整页 |
| 权限不足 | 后台路由拦截，转 403 页面，并记录访问尝试 |
| 表单校验失败 | 字段级错误 + 顶部摘要 + 自动滚动定位 |
| 数据字段缺失 | 对比页标记“资料不足”，不伪造数据 |
| 媒体上传失败 | 保留本地缩略预览，提示重新上传 |
| AI 回答缺少上下文 | 提示“资料不足，建议补充字段”，展示缺失项 |

### 5.5 交互说明文档输出要求

前端交互设计最终必须包括：

1. 页面流转图
2. 核心场景泳道图

5. 组件交互说明

## 6. 前后端接口对接文档

### 6.1 API 分组

接口分为公开接口和后台接口两组：

- 公开接口：`/api/public/*`
- 后台接口：`/api/admin/*`

### 6.2 前端接口调用层设计

```text
src/lib/api/
  client.ts
  interceptors.ts
  errors.ts
  cache-policy.ts
  public/
  admin/
```

#### 请求封装要求

1. 所有请求统一通过 `apiClient` 发出。
2. 自动拼接基础域名、语言、追踪头。
3. 支持请求取消、超时控制和幂等重试。

#### 响应拦截要求

1. 统一拆解 `data / meta / error` 结构。
2. 401 自动清理后台会话并跳转登录页。
3. 403 进入无权限页并记录来源页面。
4. 422 将后端字段错误映射到前端表单。
5. 500 系统错误展示可重试页和错误编号。

#### 错误统一处理

| 错误类型 | 处理策略 |
| --- | --- |
| 401 未登录 | 跳登录，保留 returnUrl |
| 403 无权限 | 拦截访问并展示权限说明 |
| 404 资源不存在 | 跳转实体级 404 页面 |
| 409 数据冲突 | 提示重新获取最新版本 |
| 422 字段校验失败 | 表单错误映射 |
| 5xx 服务异常 | 全局错误边界 + 上报日志 |

#### 缓存策略

| 数据类型 | 缓存方式 | 策略 |
| --- | --- | --- |
| 首页与详情公开内容 | Next ISR + Query 缓存 | 高频内容 10 分钟增量刷新 |
| 搜索结果 | Query 缓存 | 按关键词和筛选条件缓存 3 分钟 |
| 对比结果 | Query 缓存 | 按机型组合缓存 5 分钟 |
| 后台列表 | Query 缓存 | 进入页命中缓存，保存后失效 |
| 后台表单草稿 | 本地缓存 | 防刷新丢失 |

### 6.4 AI 接口接入设计

问答能力使用独立服务层封装，前端侧只关注上下文和渲染：

1. 页面在打开问答抽屉时传入 `scene`、`entityType`、`currentEntity`、`comparisonEntities` 等上下文。
2. 响应结构统一渲染为主回答、重点摘要、推荐项、缺失字段。
3. 当后端返回 `missingFields` 时，公开页展示“资料不足”，后台页展示“建议补全”。

## 7. 性能指标规划

### 7.1 核心指标目标

| 指标 | 目标值 | 适用页面 |
| --- | ---: | --- |
| 首屏可见时间 | <= 2.5s | 首页、搜索页、详情页 |
| LCP | <= 2.5s | 首页、详情页 |
| INP | <= 200ms | 搜索、筛选、抽屉、表单 |
| CLS | <= 0.1 | 全站 |
| JS 首屏包体 | <= 220KB gzip | 首页 |
| 图片首屏总量 | <= 600KB | 首页与详情页首屏 |

### 7.2 性能优化路径

1. 公开内容使用服务端渲染 + ISR。
2. 非首屏模块采用懒加载，如知识图谱、时间轴、问答抽屉。
3. 图片使用现代格式和响应式尺寸裁剪。
4. 对比表采用虚拟滚动或局部渲染，避免一次性渲染超大矩阵。
5. 查询结果采用本地缓存和请求去重。
6. 动效只作用于关键区块，避免大面积重排和滤镜滥用。

### 7.3 性能观测

1. 开发阶段接入 Lighthouse CI。
2. 线上阶段接入 `web-vitals` 上报。
3. 对首页、详情页、搜索页建立性能看板。
