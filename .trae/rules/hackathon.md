# 赛题三竞赛规则

## 一、赛题名称

**赛题三：飞行者图鉴——开放型航空科普知识平台**

## 二、竞赛宗旨与目标

本赛题面向航空科普、数字教育与知识平台建设场景，鼓励参赛团队开发一款兼具知识展示、信息检索、多维对比与内容管理能力的航空科普教育平台 Demo。竞赛目标如下：

1. 围绕航空器、航空大事件、航空人物等主题，构建结构清晰、交互友好、可持续扩展的知识平台产品。
2. 提升公众尤其是非专业用户对航空科技、航空史与飞行器知识的理解。
3. 鼓励作品兼顾专业性、教育性、可用性与创新性，形成可演示、可评审、可延展的数字化成果。
4. 综合考察参赛团队在产品设计、技术实现、数据组织、交互体验、内容策划与现场表达等方面的能力。

## 三、参赛对象与组队要求

1. 参赛对象为高校学生、开发者、设计师、产品经理、科普工作者及其他对航空科普数字化建设感兴趣的个人或团队。
2. 每支参赛队伍人数为 1 至 5 人，每名参赛者仅可加入 1 支队伍，不得重复参赛。
3. 每支队伍须指定 1 名队长，负责报名、材料提交、联络确认、答辩组织与规则接收。
4. 鼓励跨学科组队，建议团队至少包含产品策划、前端开发、后端或数据、视觉或交互等角色。
5. 若主赛事对参赛资格、年龄、身份或单位属性另有规定，则以主赛事总章程为准。

## 四、赛题任务说明

参赛作品须围绕“航空科普知识平台”展开，至少包含前端展示界面与简易管理后台两部分。

### 4.1 前端功能要求

前端部分至少应支持以下能力：

1. 浏览并搜索航空器、航空大事件、航空人物三类内容。
2. 对任意多款航空器进行横向对比，对比维度至少包含外观、尺寸、速度、航程、发动机类型、首飞年份等核心参数。
3. 通过结构化页面展示科普信息，使非专业用户能够理解相关知识。

### 4.2 后台功能要求

后台部分至少应支持以下能力：

1. 新增、编辑、删除航空器等内容条目。
2. 上传文本、图片及关键参数，并同步更新前台展示。
3. 对数据字段进行基本校验，确保内容结构完整。

### 4.3 鼓励拓展方向

在满足基础要求的前提下，参赛团队可增加以下创新功能：

1. 时间轴展示。
2. 知识图谱或关联网络。
3. 智能问答与推荐系统。
4. 数据可视化看板。
5. 主题策展与教育引导模块。

## 七、作品提交要求

1. 提交时间以赛事平台服务器记录为准，逾期提交不予受理。
2. 每支队伍须按要求一次性提交完整材料，原则上截止后不得替换核心内容；如组委会允许补正，仅限格式性补充。
3. 必交材料包括：
   - 项目说明文档：PDF 格式，文件名统一为 `队伍名称-项目说明书.pdf`
   - 项目演示文稿：PPT 或 PDF 格式，文件名统一为 `队伍名称-答辩材料.pptx` 或 `队伍名称-答辩材料.pdf`
   - 演示视频：MP4 格式，时长 3 至 5 分钟，建议不超过 500MB
   - 源代码或仓库地址：提供压缩包或可访问仓库链接；如为私有仓库，须在评审期内授予组委会访问权限
   - 在线演示地址：如有部署版本，须提供访问链接及测试账号
   - 作品信息表：填写作品简介、功能清单、使用技术、创新点、团队分工、原创声明等
4. 项目说明文档至少应包含以下内容：
   - 项目背景与目标用户
   - 核心功能说明
   - 系统架构或技术方案
   - 数据结构设计与后台管理说明
   - 创新点与应用价值
   - 使用说明与演示路径
   - 团队分工与原创声明
5. 如作品使用开源框架、第三方模型、图片、图标、数据集或接口，须在说明文档中如实列明来源、许可证类型及使用范围。
6. 提交渠道为 `[赛事官网或报名系统或指定邮箱或云盘链接]`；如采用邮箱提交，邮件主题统一为 `赛题三-队伍名称-作品提交`。

## 八、作品内容与格式规范

1. 作品须紧扣赛题三主题，不得偏离航空科普知识平台核心方向。
2. 作品应具备基本可运行性，至少完成主要功能流程的可演示闭环。
3. 前端界面应支持基础交互操作，内容展示完整，排版清晰可读。
4. 后台功能应具备基本内容管理能力，不得仅以静态页面替代后台。
5. 演示视频须清晰展示核心功能，不得仅展示概念海报或静态原型图。
6. 答辩材料应与提交作品一致，不得临时展示未提交版本中的核心功能。
7. 所有提交文件应使用规范命名，不得包含无关广告、恶意程序、木马、后门或其他危害评审环境的内容。

## 九、评审机制

1. 竞赛评审分为初赛材料评审与决赛现场或线上答辩评审两个阶段。
2. 初赛重点审查作品完成度、主题契合度、技术实现合理性与基础创新性。
3. 决赛重点审查作品演示效果、答辩表现、应用价值、内容质量与综合表现。
4. 评审工作由组委会邀请的技术专家、行业专家、教育专家及产品专家共同完成。
5. 评审实行独立评分、汇总计算、集中复核机制，确保评分过程客观公正。
6. 如评委与参赛队存在指导、雇佣、合作或其他可能影响公正性的关系，应主动申请回避。

## 十、评审标准与权重分配

| 评审维度 | 权重 | 评审要点 |
| --- | ---: | --- |
| 主题契合度 | 10% | 是否紧扣赛题三目标，是否完整覆盖航空器、事件、人物等核心知识场景 |
| 功能完整度 | 20% | 是否具备前台展示、搜索浏览、飞行器对比、后台管理等基础闭环 |
| 技术实现与工程质量 | 20% | 架构合理性、系统稳定性、代码组织、数据结构设计、可扩展性 |
| 内容质量与专业性 | 15% | 数据准确性、信息结构化程度、科普表达清晰度、资料来源可靠性 |
| 交互体验与视觉设计 | 15% | 界面美观、交互流畅、信息层级清晰、用户易用性 |
| 创新性 | 10% | 在展示方式、智能能力、知识组织、交互设计等方面的创新程度 |
| 应用价值与可推广性 | 5% | 面向教育、科普、展陈或公共传播的实际价值 |
| 答辩与展示表现 | 5% | 表达清晰、演示顺畅、回答问题准确、团队协作良好 |

## 十一、评分细则

1. 评分采用百分制，按各维度加权求和计算总分。
2. 分数区间说明如下：
   - 90 至 100 分：明显优于平均水平，功能完整，赛题契合度高，技术与内容质量突出，演示流畅，具备较强推广价值。
   - 80 至 89 分：完成度较高，主要功能实现完整，设计与技术表现较好，存在少量可优化问题但不影响总体质量。
   - 70 至 79 分：基本完成赛题要求，具备核心功能与可演示成果，但在创新性、稳定性、内容深度或体验方面存在明显短板。
   - 60 至 69 分：仅满足部分要求，功能不完整或实现较粗糙，后台、对比、搜索等关键模块缺失或质量较弱。
   - 60 分以下：未达到竞赛基本要求，主题偏离明显，无法正常演示，或存在合规问题。
3. 如出现总分并列，按以下顺序比较：
   - 功能完整度得分
   - 技术实现与工程质量得分
   - 内容质量与专业性得分
4. 如上述分项仍完全相同，由评审组集体复议决定最终排名。

## 十三、合规性要求

1. 参赛作品必须由参赛团队独立完成，不得抄袭、剽窃、套用他人作品，不得侵犯任何第三方知识产权、肖像权、隐私权或其他合法权益。
2. 参赛团队不得通过恶意刷票、伪造数据、冒名参赛、篡改评审材料、伪造开发记录、操纵演示环境等方式获取不正当竞争优势。
3. 参赛团队不得提交与赛题无关或重复参加其他组别评审的同一作品，除非主办方明确允许。
4. 允许合理使用开源框架、公开数据集和通用第三方服务，但必须遵守其许可证与使用条款，并进行明确标注。
5. 如作品中使用生成式 AI 辅助开发、文案生成、图像生成或代码生成，须如实说明使用范围，不得将 AI 自动生成内容虚假陈述为完全人工原创成果。
6. 参赛作品不得含有违法违规、政治敏感、色情低俗、暴力血腥、恶意攻击、歧视性或其他不适宜公开展示的内容。

## 十四、违规认定与处罚规则

1. 经查实存在抄袭、剽窃、作弊、恶意攻击评审系统、伪造材料或其他严重违规行为的，组委会有权直接取消参赛资格或获奖资格。
2. 如违规行为发生在评审前，取消报名或参赛资格。
3. 如违规行为发生在评审后但奖项公布前，取消入围或获奖资格，成绩作废。
4. 如违规行为在获奖公布后被查实，组委会有权撤销奖项、追回奖金和证书，并予以通报。
5. 因作品侵权或违法违规引发的全部法律责任，由参赛团队自行承担；因此给组委会、合作单位或第三方造成损失的，参赛团队应依法承担赔偿责任。
6. 对轻微且可补正的格式性问题，组委会可给予一次限时补正机会；逾期未补正的，视为放弃或按不合格处理。

## 十五、知识产权归属

1. 参赛作品的著作权及其他相关知识产权原则上归参赛团队所有。
2. 参赛团队提交作品即视为同意授予组委会及主办方非独占、免费的展示、传播、汇编、宣传权利，用于赛事宣传、成果展示、媒体报道、案例汇编与公益推广。
3. 未经参赛团队书面同意，组委会不得将参赛作品用于与赛事无关的商业开发；如涉及后续孵化、落地合作或商业转化，应另行签署合作协议。
4. 参赛团队应确保其提交的代码、图片、视频、字体、数据及其他素材具有合法来源；因权属不清引发纠纷的，由参赛团队自行负责。
5. 如作品中含有第三方受限资源、商业接口或不可公开代码，团队须在提交时明确说明评审可访问方式与使用边界。

## 十六、答辩规则

1. 入围决赛团队须按组委会通知参加答辩，无故缺席视为自动放弃。
2. 每支队伍答辩时间建议为 8 至 12 分钟，其中作品展示 5 至 8 分钟，评委问答 3 至 4 分钟。
3. 答辩内容应包括项目背景、目标用户、核心功能、技术方案、创新点、应用价值及现场演示。
4. 决赛展示内容须与已提交作品一致，不得使用与提交版本明显不一致的核心能力替代参评。
5. 若因现场网络或设备问题影响演示，团队可使用预录视频作为辅助说明，但仍需接受评委问答。

## 十七、申诉与复核机制

1. 参赛团队如对评审流程、资格认定或违规处理存在异议，可在结果公布后 24 小时内向组委会提交书面申诉。
2. 申诉材料应包括队伍名称、联系人、申诉事项、事实依据及相关证明材料。
3. 组委会将在收到申诉后 3 个工作日内完成复核并反馈结果。
4. 对评委基于专业判断作出的正常评分意见，原则上不接受针对主观分值高低的单独申诉；仅对程序性、公平性、资格性问题进行复核。

## 十八、突发情况处理机制

1. 如遇自然灾害、公共卫生事件、重大网络故障、平台宕机、电力中断或其他不可抗力，组委会有权调整赛程、提交方式、评审方式或答辩形式，并及时公告。
2. 如赛事平台在截止前发生大面积故障，组委会可适当顺延提交截止时间，具体以公告为准。
3. 如参赛队伍因突发情况无法按时参赛，应在第一时间向组委会报备并提供必要说明；是否准予延期或变更，由组委会根据实际情况决定。
4. 如答辩现场发生网络异常、设备故障等影响正常展示的情况，组委会可安排补充演示、延后答辩或采用备用方案。
5. 任何突发情况处理均以保障赛事公平、确保评审一致和维护大多数参赛队伍合法权益为原则。
